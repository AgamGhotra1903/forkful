================================================================================
                      KEY ARCHITECTURAL DECISIONS
================================================================================

✅ FULL-SCREEN MAPS
   Why: Matches modern UX (Uber-like), maximizes usable area
   Trade-off: Overlaying bottom sheets instead of side-by-side
   Solution: Transparent panels, proper z-indexing

✅ LEAFLET INSTEAD OF GOOGLE MAPS
   Why: Free (no API key), uses OpenStreetMap, privacy-friendly
   Trade-off: Less detailed satellite imagery
   Solution: CartoDB dark tiles provide excellent UX

✅ OSRM ROUTING INSTEAD OF COMMERCIAL SERVICES
   Why: Completely free, no API rate limits
   Trade-off: Slightly less accurate than Mapbox/Google
   Solution: Falls back to straight line if unavailable

✅ REDIS ADAPTER (NOT REQUIRED FOR SINGLE INSTANCE)
   Why: Future-proof for scaling with minimal overhead
   Trade-off: Additional dependency (Redis)
   Solution: Gracefully degrades if Redis unavailable

✅ IP_HASH STICKY SESSIONS
   Why: Socket.io works best with sticky sessions
   Trade-off: Less perfect load balancing
   Solution: Acceptable for delivery app (long-lived connections)

✅ SOCKET.IO OVER REST
   Why: Real-time updates without polling
   Trade-off: Stateful connections
   Solution: Redis adapter handles multi-instance

✅ MICROSERVICES OVER MONOLITH
   Why: Independent scaling, easier maintenance
   Trade-off: Increased complexity
   Solution: Docker orchestration simplifies deployment
